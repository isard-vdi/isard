# GPU management in webapp

Human-readable functional specification of the **create**, **edit**,
**delete**, **force active profile**, and **enable/disable profile**
flows for a reservable GPU from the legacy admin. Serves as the
contract for the E2E test `tests/webapp/gpus.spec.js`.

## Scope

- **Component**: admininistration panel.
- **Screen**: **GPUs** section inside the **Hypervisors** view. The
  **Bookables** screen is secondary: we check side effects of the
  profile flow there.
- **Actions covered**:
  - Create a new GPU tied to an existing GPU profile.
  - Edit the name and description of an existing GPU.
  - Delete a GPU (with confirmation, plus an extra dialog if it has
    dependencies — desktops, plans, bookings, deployments).
  - Force the active profile of a physical card.
  - Enable and disable individual profiles of a GPU.
- **Out of scope**: reservation and assignment of the GPU to
  desktops/deployments, time-window planning.

## Common role and prerequisites

| Element | Expected value |
| --- | --- |
| Role | Administrator of the `default` category |
| Session | Logged in to the webapp |
| Infrastructure | At least one registered hypervisor with an available GPU profile |
| GPU list | The GPU table has loaded and is visible |

## Common data

| Field | Sample value | Notes |
| --- | --- | --- |
| Name | `e2e-gpu-<worker>-<timestamp>` | 4–50 chars; letters, digits, space, `.`, `-`, `_`, and accents |
| Description | `e2e GPU created at <ISO timestamp>` | Free text |
| GPU profile | The first profile available in the dialog | Picked from the system list |

The GPU name is stored in the test metadata so that `afterEach` can
recover and clean it up even if assertions fail.

## Scenario 1 — *admin creates a GPU and sees it listed*

### Given

1. The administrator is authenticated in the webapp.
2. They are on the **Hypervisors** page and the **GPUs** section table
   has loaded.

### When

1. They press the **Add new GPU** button in the GPUs section.
2. The GPU creation dialog opens.
3. They fill in the **name** and **description**.
4. They pick a **GPU profile** from the list shown in the dialog.
5. They confirm the creation (**Send** button).

### Then

1. The system stores the GPU. The API call
   `POST /api/v4/item/reservable/gpus` responds with status `< 400` and
   the body is `SimpleResponse(id=<new_gpu_id>)` — the id generated by
   RethinkDB comes back in the body, so no follow-up lookup is needed
   to resolve it.
2. The dialog closes automatically.
3. The new GPU appears as a row in the GPUs table, showing the entered
   name and description.
4. If the GPU is queried via API (`GET /api/v4/items/reservables/gpus`),
   it is present with the same `id` returned by the POST, and with the
   expected name and description.

## Scenario 2 — *admin edits a GPU's name and description*

### Given

1. A GPU created by this test exists (see Scenario 1) and is visible in
   the GPUs table.

### When

1. On the GPU row, they press the **edit** icon (pencil).
2. The edit dialog opens with the current **name** and **description**
   pre-filled.
3. They change the name to a new value (`<original>-edited`) and the
   description to new text.
4. They confirm the changes (**Send** button).

### Then

1. The system saves the changes. `PUT /api/v4/admin/reservables/gpus/{id}`
   responds with status `< 400` and the body is
   `SimpleResponse(id=<item_id>)` (path echo).
2. The dialog closes automatically and an update-success notification
   appears.
3. The GPU row in the table refreshes and shows the new name and
   description.
4. If the GPU is queried via API, it returns the updated values.

---

## Scenario 3 — *admin deletes a GPU without dependencies*

### Given

1. A GPU created by this test exists, **with no** associated desktops,
   plans, bookings, or deployments.

### When

1. On the GPU row, they press the **delete** icon (red cross).
2. A confirmation dialog appears ("Do you really want to delete this
   gpu?").
3. They confirm the action.

### Then

1. The system checks the GPU has no dependencies and proceeds straight
   to deletion (no extra dialog).
2. A deletion-success notification appears.
3. The row disappears from the GPUs table.
4. If the GPU is queried via API, it is no longer there.

## Scenario 4 — *admin deletes a GPU with dependencies and notifies users*

> **E2E test status**: `test.skip` — the dev DB seed does not ship a
> GPU with the full quartet desktop + plan + booking + deployment all
> tied to the same GPU, and synthesising it requires the bookings and
> deployments APIs which fall outside the scope of this spec. The
> *disable with dependencies* branch is covered in Scenario 7 via the
> seed `e2e8b73f-…` + `NVIDIA-A16-4Q`.

### Given

1. A GPU exists with at least one desktop or plan using it.

### When

1. On the GPU row, they press the **delete** icon (red cross).
2. They confirm the initial dialog.
3. A second dialog opens listing the affected desktops, plans,
   bookings, and deployments, and offering the **Notify user**
   checkbox.
4. They check **Notify user** and press **Send**.

### Then

1. The system deletes the GPU and notifies the affected users.
2. The dialog closes and a deletion-success notification appears.
3. The row disappears from the GPUs table.
4. The affected desktops/plans receive the corresponding notification.

---

## Scenario 5 — *admin enables a GPU profile*

### Given

1. A GPU exists with at least one **disabled** profile.

### When

1. In the GPUs table, expand the GPU row to see its profiles.
2. Tick the checkbox of a profile that was disabled.

### Then

1. The system enables the profile immediately.
   `PUT /api/v4/item/reservable/enable/gpus/{item_id}/{subitem_id}`
   responds with status `< 400` and the body is
   `SimpleResponse(id=<item_id>)`.
2. The checkbox stays checked and persistence is validated via
   `POST /api/v4/admin/table/gpus` (the `profiles_enabled` field must
   contain the submitted `subitem_id`).
3. If this was the first time this profile was enabled in the system,
   the matching bookable appears in the **Bookables** table
   (`reservables_vgpus`) — covered separately in Scenario 13.
4. If other GPUs already had the same profile enabled, the existing
   bookable is reused and its `total_units` is updated to account for
   the new GPU.

---

## Scenario 6 — *admin disables a GPU profile without dependencies*

### Given

1. A GPU exists with an **enabled** profile and no desktops or plans
   using it.

### When

1. Expand the GPU row.
2. Uncheck the profile checkbox.

### Then

1. The system checks the profile is not the "last" with dependencies
   and disables it directly.
2. A "GPU profile disabled" notification appears.
3. **Important side effect** — the backend re-counts how many GPUs
   have this profile enabled:
   - If other GPUs still have the profile enabled, the bookable stays
     in **Bookables** (`reservables_vgpus`) with recalculated
     `total_units`.
   - If this was the **last** GPU in the system with this profile
     enabled, the bookable is **deleted** automatically from
     `reservables_vgpus` and disappears from the Bookables table in
     the webapp.

## Scenario 7 — *admin opens the dependencies dialog and cancels it*

> The E2E test runs against the seeded pair
> (`e2e8b73f-b989-47b4-9864-9e0da97f7b21`, `NVIDIA-A16-4Q`) — the only
> seed combination with desktops referencing it and no other GPUs
> enabling the same profile (`last = [true]`). To avoid races between
> parallel workers on this shared state, the test presses **Cancel**;
> the *Send + Notify user* branch is covered by apiv4 unit tests.

### Given

1. An enabled profile of a GPU has at least one desktop or plan using
   it, and on this GPU it is the **last** one enabled for that
   profile.

### When

1. Expand the GPU row.
2. Uncheck the profile checkbox.
3. The "Disable profile" dialog opens, listing affected desktops,
   plans, bookings, and deployments, with the **Notify user**
   checkbox.
4. Press **Cancel**.

### Then

1. The call `GET /api/v4/item/reservable/check-last/gpus/{subitem_id}/{item_id}`
   responds with status `< 400` and its response contains
   `last = [true]` and a non-empty desktops list.
2. The "Disable profile" dialog is shown with the affected desktops
   table populated (no "no items" placeholder).
3. On Cancel, the dialog closes, the profile checkbox **becomes
   checked again** (`profile_checkbox.prop("checked", true)`) and **no**
   `PUT /api/v4/item/reservable/enable/...` call is made.
4. The seeded state is preserved: the GPU still has the profile in
   `profiles_enabled` when queried again.

### *Send + Notify user* path (not covered by the E2E test)

If the admin confirms with **Notify user** checked:

1. The system disables the profile and notifies the affected users.
2. The profile checkbox stays unchecked and the "GPU profile disabled"
   notification appears.
3. If this was the last GPU in the system with this profile enabled,
   the bookable is deleted from `reservables_vgpus` (disappears from
   Bookables). Otherwise only `total_units` is updated.

## Scenario 8 — *admin forces the active profile of a GPU*

> In the E2E test this scenario splits into two error cases (**S8a**
> and **S8b**). The success path is not covered: the only seed with
> `physical_device` + `active_profile` populated is
> `e2e-gpu-force-a40`, which serves S8b but cannot be used to exercise
> a transition to a different profile without affecting state shared
> with other tests.

### Scenario 8a — *force button with no enabled profiles*

#### Given

1. A freshly created GPU exists with `profiles_enabled = []` (no
   profile enabled yet).

#### When

1. On the GPU row, press the **force** icon (lightning).

#### Then

1. The page JS detects `profiles_enabled.length === 0` and shows a
   PNotify "No profiles enabled" notification.
2. The `#modalForcedProfile` dialog does **not** open.

### Scenario 8b — *force the profile that is already active*

#### Given

1. A seeded GPU exists (`e2e-gpu-force-a40`) with `physical_device`,
   `active_profile = "2Q"`, and `NVIDIA-A40-2Q` in
   `profiles_enabled`.

#### When

1. On the GPU row, press the **force** icon (lightning).
2. The **Forced profile** dialog opens with the dropdown populated
   from `POST /api/v4/admin/table/gpus` with only the enabled
   profiles.
3. Select `NVIDIA-A40-2Q` (the one already active) and press **Send**.

#### Then

1. The handler detects the chosen profile is already active and shows
   a PNotify with the text "already the active profile".
2. **No** call is made to `/engine/profile/gpu/{physical_device}`.

### Success path (not covered by the E2E test)

If the admin picks a profile **different** from the active one:

1. If the change crosses **MIG** and **vGPU/passthrough** (profiles
   starting with `<n>g.`), the browser shows a confirmation warning
   explaining the GPU will be reset and running instances stopped.
2. If confirmed, the system sends the order to the engine and responds
   with success.
3. The dialog closes and the notification "GPU active profile updated
   successfully" appears.
4. If the GPU is queried, the active profile reflects the new value.

### Other known error cases

- **Cancelling the MIG↔vGPU change confirm**: no call is made and the
  dialog stays closed.

## Scenario 9 — *admin tries to create a GPU with an invalid name*

### Given

1. The admin is on the GPU creation dialog with a profile already
   selected.

### When

1. They enter an invalid value in the **Name** field. Cases to cover:
   - Too short (less than 4 characters), e.g. `abc`.
   - Contains characters outside the allowed set
     (`^[\-_àèìòùáéíóúñçÀÈÌÒÙÁÉÍÓÚÑÇ .a-zA-Z0-9]+$`), e.g. `gpu@1`,
     `my/gpu`, `名字`.
   - Empty.
   - (The *too long* branch cannot be exercised through `fill()`
     because the `<input>` has `maxlength="50"` and the browser trims
     the value before it reaches the validator.)
2. Press **Send**.

### Then

1. Client-side validation (Parsley) blocks the submission.
2. An inline error message appears on the name field indicating the
   reason (length or pattern).
3. **No** call is made to `POST /api/v4/item/reservable/gpus`.
4. The dialog stays open and keeps the rest of the entered data.

---

## Scenario 10 — *admin confirms the creation without picking a profile*

### Given

1. The admin is on the GPU creation dialog with the name and
   description filled in with valid values.
2. They have **not** clicked any row in the dialog's profiles table.

### When

1. They press **Send**.

### Then

1. The system detects no profile is selected and blocks the
   submission.
2. A visible warning appears on the dialog ("No bookable selected" or
   equivalent) highlighting the profiles table.
3. **No** call is made to `POST /api/v4/item/reservable/gpus`.
4. The dialog stays open and keeps the entered data so the admin can
   pick a profile and retry.

---

## Scenario 11 — *admin tries to create a second GPU with an existing name*

### Given

1. A GPU with name `e2e-gpu-<worker>-<timestamp>` exists.

### When

1. The admin opens the creation dialog, enters **the same name**, a
   valid description, and picks a profile.
2. They press **Send**.

### Then

1. The creation API responds with a **conflict** error (expected 409).
2. The dialog shows an error notification with the reason returned by
   the API.
3. The dialog stays open; no new row is added to the GPUs table.
4. If queried via API, only one GPU with that name exists.

## Scenario 12 — *admin deletes a GPU with active desktops and plans*

> Extension of Scenario 4 focused on checking the resulting state of
> the dependent resources once the GPU is deleted.
>
> **E2E test status**: `test.skip` for the same reason as Scenario 4
> — the desktop + plan + booking + deployment quartet tied to the
> same GPU is required, which the current seed does not provide.

### Given

1. A GPU exists with:
   - At least one active desktop using it.
   - At least one plan depending on it.
   - At least one booking and/or deployment associated.

### When

1. The admin runs the full flow from Scenario 4 (deletion with
   dependencies dialog and **Notify user** checked).

### Then

1. The GPU disappears from the GPUs table and is no longer returned by
   the API.
2. **Plans and bookings**: cleaned up via the planner
   (`ReservablesPlannerProccess.delete_item`); references to the
   deleted GPU are invalidated and no longer appear on the **Plans**
   or **Bookings** screens.
3. **Desktops**: still exist but lose the deleted GPU's reservation;
   on next start they no longer ask for that resource.
4. **Notifications**: each affected user receives a *deleted-gpu*
   notification (email via the notifier service).
5. The **Bookables** table reflects the removal of profiles that no
   longer have any GPU in the system with them enabled (side effect of
   Scenario 6).

## Scenario 13 — *end-to-end check on the Bookables screen*

> Minimal cross-check between the **Hypervisors** screen and the
> **Bookables** screen: when enabling a GPU profile from the details
> subtable, the matching entry must appear in Bookables.

### Given

1. A GPU exists named `e2e-gpu-<worker>-<timestamp>` with no profile
   enabled (`profiles_enabled = []`).
2. Profile X to be used is known and **no other GPU** in the system
   has it enabled — the **Bookables** screen shows **no** entry for
   X.

### When

1. The admin enters the **Hypervisors** screen and locates the GPU row
   in `#table-gpus`.
2. They click the row's detail button to expand it and see the
   profiles subtable.
3. They tick the checkbox of profile X in the subtable.
4. They wait for the success notification ("GPU profile enabled").
5. They navigate to the **Bookables** screen.

### Then

1. When the checkbox is ticked,
   `PUT /api/v4/item/reservable/enable/gpus/{gpu_id}/{profile_id}` is
   called with status `< 400`.
2. A PNotify success notification with title *"GPU profile enabled"*
   is shown.
3. The GPU has profile X in `profiles_enabled` (verifiable via
   `POST /api/v4/admin/table/gpus`).
4. On the **Bookables** screen, the `#reservables_vgpus` table shows
   a new row with `id` equal to profile X.

## Cleanup (afterEach)

1. The GPU name created by this test is recovered from the metadata.
2. The GPU is looked up by name and, if it exists, deleted via API.
3. If the test changed enabled-profile state, the original state must
   be restored (re-enable/disable as needed) before deleting the GPU.
4. Cleanup errors are silenced to avoid masking the real reason of an
   earlier failure.

## APIs touched by the flows (reference)

- `POST   /api/v4/item/reservable/gpus` — create. Body
  `AddReservableItemRequest`; response `SimpleResponse(id=<new_id>)`.
- `PUT    /api/v4/admin/reservables/gpus/{id}` — edit (name and
  description). Body `UpdateReservableItemRequest`; response
  `SimpleResponse(id=<id>)`.
- `GET    /api/v4/item/reservable/check-last/gpus/{id}` — check
  dependencies before deleting the GPU. Response `CheckLastResponse`.
- `GET    /api/v4/item/reservable/check-last/gpus/{subitem_id}/{item_id}`
  — check dependencies before disabling a profile. Response
  `CheckLastResponse`.
- `DELETE /api/v4/item/reservable/gpus/{id}[?notify_user=true]`
  — delete the GPU. Response `EmptyResponse`.
- `PUT    /api/v4/item/reservable/enable/gpus/{item_id}/{subitem_id}[?notify_user=true]`
  — enable/disable a profile. Body `EnableReservableRequest`; response
  `SimpleResponse(id=<item_id>)`.
- `PUT    /engine/profile/gpu/{physical_device}` — force active
  profile (not apiv4 — engine).
- `GET    /api/v4/items/reservables/gpus` — verification and cleanup
  lookup. Response `ReservableDetailResponse`.
- `GET    /api/v4/items/reservables/profiles/gpus` — profile catalog
  for the *Add new GPU* modal. Response
  `list[ReservableProfileResponse]`.

## Relevant database state

- `gpus.profiles_enabled` table (list of profile ids enabled for each
  GPU).
- `reservables_vgpus` table (the *bookables* visible on the Bookables
  screen): **created/updated** when a profile is enabled, and
  **deleted** when its last use across the whole system is unchecked.

## Cases not covered (future)

- Force a MIG↔vGPU profile change and check the real effect on running
  instances (requires compatible infrastructure).
